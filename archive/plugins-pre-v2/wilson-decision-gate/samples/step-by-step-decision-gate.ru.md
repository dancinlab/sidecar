# Пошаговый шлюз решений — canonical-референс

> Long-form образец для sidecar-плагина **`wilson-decision-gate`** (standalone-порт governance-принципа wilson `step-by-step-decision-gate`).
> По умолчанию = **ON** при установке плагина. Выключить: `/wilson-decision-gate off`, `/sidecar off decision-gate` или `SIDECAR_NO_DECISION_GATE=1`.
> Этот файл — canonical-референс *как применять*: прочтите один раз и используйте как шаблон в начале многорешенческой задачи. `/wilson-decision-gate sample [en|ko|ja|zh|ru]` печатает его.

## Когда принцип в силе

Принцип срабатывает для **многорешенческой работы** — любой задачи с N точками ветвления, где выбирает пользователь. Примеры:

- Посадка новой спеки (решение = имя, набор falsifier, слой принуждения, формат аудита, …)
- Нетривиальный рефактор с выбором охвата (какие файлы in/out, форма абстракции, порядок миграции, окно deprecation, …)
- Дизайн API (имена глаголов, формы параметров, модель ошибок, политика версионирования, …)
- План миграции (порядок шагов, шлюзы отката, форма промежуточного состояния, …)
- Любая Plan-mode работа, где всплывает ≥ 2 жизнеспособных подхода

Не срабатывает на разовых задачах (фикс одного бага, краткий Q&A, механические правки) — там нет батчинга, который надо побеждать.

## Что значит «одно решение = один пользовательский шлюз»

Для каждой точки ветвления агент **обязан**:

1. **Подать точку ветвления отдельно** — никогда не связывать два решения в один yes/no.
2. **Перечислить варианты** (≥ 2), каждый с однострочным описанием.
3. **Рекомендовать один** (`### 🎯 추천:` или эквивалентный заголовок) с **3+ пунктами обоснования** под «**근거**:» / «**Why**:».
4. **Дождаться выбора пользователя** перед переходом к следующему решению. Используйте `AskUserQuestion`, если доступно; иначе остановитесь и спросите простым текстом.
5. **Записать выбор + обоснование** в `design.md` (или журнал сессии) по ходу — `/wilson-decision-gate decide "<выбор>" "<обоснование>"` добавит в формате ниже.

Аудит-трейл (`design.md` секция решений + перекрёстная ссылка на спеку / PR, если применимо) — **часть результата**, а не довесок.

## Почему без батчинга

Батчинг схлопывает N выборов в один yes/no. Конкретно:

- Большинство выборов проходят необдуманно — пользователь читает заголовки, говорит «да всему», нагруженные выборы штампуются.
- Перекладывание вины задним числом неизбежно — когда один из связанных выборов оказался неверным, «ты же сказал да» — хрупкий якорь аудита.
- Шлюз перестаёт быть шлюзом — становится театром подтверждения. Весь смысл step-by-step — замедлить каждую точку ветвления до её собственного решения.

Поэтому протокол, который обобщает этот принцип, задаёт жёсткое правило: **выбрано больше одного решения на шлюз → блок**.

## Формат записи решения

Внутри design.md / журнала сессии каждое решение ложится так:

```markdown
### Решение N — <однострочное описание>
- **picked**: ⭐ <ярлык варианта> (или 🥇 / ✅ / ❌ при ранжировании нескольких)
- **обоснование**:
  - <пункт 1 — почему лучше альтернатив>
  - <пункт 2 — какой компромисс принимается>
  - <пункт 3 — что позже опровергнет этот выбор>
```

Три пункта — минимум. Для несущих решений можно больше. Меньше = обоснование настолько тонкое, что будущий читатель (вы через 3 недели) не реконструирует компромисс.

## Перекрёстные ссылки

- Спека / PR, которым принадлежат решения, должны нести строку `decision_audit_ref: <путь к design.md>` (двунаправленно — спека → design.md, шапка design.md → спека/PR).
- Если ведёте журнал сессии, формат записи решения выше входит в него отдельной секцией.

## Worked example

Берите любую реальную многорешенческую сессию спеки/дизайна как worked example: каждая точка ветвления получает свой шлюз, строку `picked` и обоснование из 3 пунктов, всё перекрёстно связано со спекой/PR через единый `decision_audit_ref`. Прочтите один такой design.md, чтобы усвоить паттерн, затем переиспользуйте этот шаблон.

## Контрпример (когда НЕ применять)

- Разовый фикс бага с ясной первопричиной — нет решения для шлюза.
- Механический рефактор (переименование, форматирование, фикс линта) — нет точек ветвления.
- Q&A / объяснение / отчёты о статусе — нет результата.
- Срочный хотфикс, где скорость > обдумывания (используйте kill-switch `/wilson-decision-gate off` / `SIDECAR_NO_DECISION_GATE=1` на время сессии и запишите причину).

## Связь со стилем ответа `wilson-prefs`

Протокол, который это обобщает, связывает две вещи по естественным осям:

1. **Стиль ответа** (например, friendly 7-element паттерн, формат рекомендации, enum эмодзи) — это *как вы пишете*. Живёт отдельно в sidecar-плагине `wilson-prefs` как стиль ответа.
2. **Пошаговый шлюз подтверждения пользователем** — *этот* принцип, *как вы решаете*.

**Область независима** (любой может быть включён при выключенном другом), **но представление наследуется — автоматически**:

- Опции + рекомендация + обоснование шлюза **отображаются в том стиле ответа, который сейчас объявлен `wilson-prefs`** — агент читает активный стиль из внедрённого блока `## Prefs` (`Active response style: **<name>**`) и применяет его **без повторного запроса пользователя**. Если `wilson-prefs` уже объявил `style=friendly`, а пришлось вручную сказать «покажи шлюз в friendly-версии» — это **баг**, а не задуманный рабочий процесс.
- Когда активный стиль — **friendly**, каждая опция и рекомендация используют **полный friendly 7-element паттерн** (эмодзи-иконка · алиас · одна строка простыми словами · бытовая аналогия · fenced ASCII-диаграмма · сравнение с ближайшим инструментом) — *не* bare terse table.
- Когда `wilson-prefs` **отсутствует / не задан / отключён**, шлюз откатывается к представлению хоста по умолчанию. Жёсткой рантайм-связи нет — шлюз работает автономно, просто не может унаследовать стиль, который никогда не задавался.

Коротко: *шлюзовать ли* — это данный принцип; *как читается шлюз* — наследуется от активного стиля prefs, без ручного напоминания.

## Шпаргалка активации

```sh
# по умолчанию: плагин ON после установки (SessionStart внедряет принцип;
# UserPromptSubmit добавляет короткое напоминание только на промптах,
# похожих на точку ветвления — не на каждом промпте).

/wilson-decision-gate off          # выключить внедрение принципа
/wilson-decision-gate on           # снова включить
/sidecar off decision-gate         # рантайм-переключение через control-плагин
SIDECAR_NO_DECISION_GATE=1         # kill switch на сессию (env, всегда главнее)

/wilson-decision-gate decide "<выбор>" "<обоснование>"   # добавить Decision
/wilson-decision-gate log          # показать секцию решений design.md
/wilson-decision-gate path <file>  # указать другой design.md
```
