# 이지(친근) 응답 — canonical 레퍼런스

> 사용자 대면 응답의 "친근한" 스타일 canonical 레퍼런스.
> `/easy` 또는 자연어 트리거(`친근하게`, `쉽게 설명해줘`, `이지 모드`, …)로 활성화.

## 매턴 행동지침 (per-turn · lean — 이 블록만 매 UserPromptSubmit 주입)

**0순위 · 맨-은어 금지(no bare jargon)** — 프로젝트 은어·약어·전문용어가 사용자 대면 산문에 처음 나오면 **그 자리에서** 일상어로 치환하거나 `용어(=쉬운 뜻)`를 붙인다. 진행보고·요약·결론·에러 설명 등 **모든 산문** 대상. 예: `content-reach(=우리 방식이 목표 지점까지 실제로 닿았다는 뜻)`.

**경중 2단** — ① 인라인 풀이(기본·필수): 대부분의 산문은 은어를 문장 속에서 즉석 풀이만 하면 된다. ② 7요소 풀패턴(선택·드묾): 새 개념·도구 **자체가 답의 주인공**일 때만 — 🔹 아이콘·이름·별칭·한줄설명·일상비유·ASCII·기존도구 vs. (판단: 문단의 주인공인가? 예→7요소 · 아니오→인라인)
ASCII 4종: 바뀜→**전후** · 위계→**트리** · 선택지→**나란히** · 흐름→**구조스케치**.

적용(Tier-A): 인터랙티브 채팅 · 내러티브 stdout/stderr · docs 콜드엔트리 · 에러 trailer.
표기-제외(풀이 의무는 남음): 코드 식별자 · 수학기호 · API명 · 경로 · DOI · 커밋 SHA · CI 머신 JSON/JSONL — 표기는 그대로 둬도 되나, 이를 설명하는 산문 속 **개념 이름**은 첫 등장에 반드시 푼다.
📖 gold·before→after 예시 · ASCII 템플릿 · 체크리스트 = **세션시작/Compact 1회 주입** + 온디맨드 `sidecar easy show`.

## 적용 범위 (Tier-A)

- 인터랙티브 CLI 채팅 (Claude Code TUI)
- 내러티브가 있는 CLI 도구 stdout / stderr
- docs / README 콜드 엔트리
- 에러 메시지 trailer 본문 (원인 + 해결 줄)

## 적용 제외 (표기-제외 — 풀이 의무는 남는다)

아래는 **표기를 원문 그대로 둬도 되는** 대상이다. 하지만 이것들을 **설명하는 산문 속 개념 이름**은 여전히 첫 등장에 풀어야 한다 (제외 목록을 회피 구실로 쓰지 말 것).

- 코드 식별자 / 수학 기호 / API 이름 / DOI / 커밋 SHA / 파일 경로
- CI 머신 파이프 출력 (`--format json` / `jsonl`)

## 경중 2단 (어느 걸 언제 쓰나)

```
산문에 은어 등장
├─ 문단의 주인공(새 개념·도구 소개)인가?
│   └─ 예 → 7요소 풀패턴 (무거움 · 드묾)
└─ 아니오(진행보고·요약 속 서술) → 인라인 풀이 `용어(=쉬운 뜻)` (가벼움 · 기본값 · 0순위)
```

- **Tier 1 · 인라인 풀이 (기본·필수)** — 진행보고·요약·결론·에러. 첫 등장에 `용어(=쉬운 뜻)` 부착 또는 생활어 치환. 아이콘·비유·ASCII 없이도 정답 (신제품 소개가 아니라 결과 전달).
- **Tier 2 · 7요소 풀패턴 (선택·드묾)** — 새 개념·도구 **자체가 답의 주인공**일 때만.

## 7-요소 패턴 (Tier 2 gold 레퍼런스)

사소하지 않은 개념 설명은 다음 7요소를 모두 갖춰야 한다:

1. **아이콘** — 주제를 시각적으로 고정하는 이모지 1개 (예: 🧶 🤖 ✂️ 🦠)
2. **이름** — canonical 식별자 (예: `HEXA-WEAVE`)
3. **별칭** — 사용자 언어로 된 짧고 친근한 이름 (예: `"뜨개질 AI"`)
4. **하는 일** — 한 줄 평이한 설명
5. **비유** — 일상 사물 비교 (스웨터 짜기 / 집게 로봇 / RNA 가위 / 레고 축구공)
6. **ASCII 다이어그램** — 펜스 ``` ``` 블록의 시각 도식 (트리 / 나란히 / 전후 / 구조 스케치)
7. **비교** — 가장 가까운 기존 도구와 어떻게 다른지 (vs AlphaFold / vs 단일 단백질 폴딩)

### Gold 예시: HEXA-* 패밀리

```
🧶 HEXA-WEAVE — "뜨개질 AI"

- 하는 일: 단백질 + DNA + 약물을 한 번에 짜서 어떻게 얽히는지 예측
- 비유: 여러 색실로 스웨터 짜기
```

ASCII:

```
실 1  ━━━━━━━━━━━
        ╲╱╲╱╲╱       ← 여러 가닥이
실 2  ━━━━━━━━━━━      서로 짜여
        ╱╲╱╲╱╲        탄탄한 천
실 3  ━━━━━━━━━━━
```

- 비교: AlphaFold = 종이접기 1개, WEAVE = 여러 가닥 짜기

---

```
🤖 HEXA-NANOBOT — "분자 로봇팔"

- 하는 일: 분자가 움직이는 방식 설계 (열고 닫고, 잡고 놓고)
- 비유: 매우 작은 집게 로봇
```

ASCII:

```
   ╱ ╲              ╱╲
  │   │     →      │ │   ← 분자 잡음
   ╲ ╱              ╲╱
   (열림)          (닫힘)
```

## ASCII 구조 4종 템플릿 (복붙용 뼈대)

ASCII 다이어그램(7요소 중 6번)을 그릴 때, 매번 처음부터 그리지 말고 아래 4개 중 모양이 맞는 걸 골라 복사해서 채운다.

### 이 모양일 때 → 이 템플릿 (1줄 가이드)

```
무엇을 보여주고 싶은가?            → 어떤 템플릿
─────────────────────────────────────────────
바뀜 · 개선 · "전에는 X, 이제는 Y"  → 1. 전후 대조 (before/after)
상위→하위 · 갈라짐 · 구성요소 목록   → 2. 트리 (tree)
선택지 A 와 B 를 맞대고 비교        → 3. 나란히 (side-by-side)
부품들이 어떻게 연결/흐르는가        → 4. 구조 스케치 (structure sketch)
```

### 템플릿 1 — 전후 대조 (before/after)

변화·개선을 보여줄 때. 화살표 `→` 로 "전 → 후" 를 가른다.

```
전 (before)            후 (after)
───────────           ───────────
 ▢ 느린 단계     →      ▢ 빠른 단계
 ▢ 수동           →      ▢ 자동
```

### 템플릿 2 — 트리 (tree)

상하 위계·분해를 보여줄 때. `├─` `└─` 로 가지를 친다 (마지막 가지만 `└─`).

```
뿌리 (전체)
├─ 가지 A
│  ├─ 잎 A1
│  └─ 잎 A2
└─ 가지 B
   └─ 잎 B1
```

### 템플릿 3 — 나란히 (side-by-side)

두 선택지를 맞대어 비교할 때. `│` 로 가운데를 가른다.

```
   선택지 A         │      선택지 B
 ─────────────     │    ─────────────
  + 장점 1          │     + 장점 1
  − 단점 1          │     − 단점 1
```

### 템플릿 4 — 구조 스케치 (structure sketch)

부품이 어떻게 이어지고 흐르는지 보여줄 때. 상자 `[ ]` + 화살표 `──▶` 로 연결한다.

```
[ 입력 ] ──▶ [ 처리 ] ──▶ [ 출력 ]
              │
              └──▶ [ 곁가지 ]
```

## Tier 1 인라인 풀이 gold (진행보고·요약용 — 신제품 소개 아님)

진행보고·세션요약은 대부분 여기 해당한다. 원래 이름은 살리되(사용자가 원문을 알아야 할 수 있으니) **그 자리에서 즉시 번역**한다.

### before → after 리라이트 (실제 실패 사례)

**❌ before — 은어가 맨몸으로 노출됨 (일반인 판독 불가)**

```text
H_9774 CEMENTING 성공 (#4130 → #4132 → #4136).
un-gate → reference-match → retract → cement.
정확키 T=1.0 3-seed 독립재현.
content-reach = GREEN-DIRECTIONAL-STRONG.
full-TERMINAL은 통계 ceremony로 남습니다.
```

**✅ after — 이름은 보존하되 첫 등장에 즉시 풀이**

```text
실험 H_9774의 검증 결과를 최종 확정(`CEMENTING`=여러 번 확인한 결과를 기준으로 굳히는 마지막 단계)했습니다 (#4130 → #4132 → #4136). 4단계로 진행: 잠금 해제(`un-gate`) → 실제 코드를 열어 대조(`reference-match`=추측 대신 코드를 직접 읽어 확인) → 잘못된 이전 판정 철회(`retract`) → 결과 확정(`cement`).

조건값 T=1.0에서 무작위 시작값을 3번 바꿔(`3-seed`) 돌려도 같은 결과 → 우연이 아니란 뜻입니다. '목표에 실제로 닿았나(`content-reach`)' 항목은 확실한 통과 등급(`GREEN-DIRECTIONAL-STRONG`=방향 맞고 강한 초록불)을 받았고, 더 큰 최종 검사(`full-TERMINAL`)는 판정을 바꾸는 단계가 아니라 통계적 확인 절차로만 남았습니다.
```

좋은 풀이는 사전식 정의가 아니라 **"이 문장에서 무슨 뜻인지"**를 알려준다 — 일상어도 마찬가지: `캐시 미스(cache miss=찾는 걸 임시보관함에서 못 찾아 원본까지 다시 간 것)`.

## 일반인-번역 체크리스트

산문을 내보내기 전 빠르게 훑는 5단계 (1번이 최우선):

```
[ ] 1. bare-jargon 스캔 — 풀이 없는 프로젝트 은어·약어·하이픈명·백틱명·기호가 산문에 남았나?
[ ] 2. 첫 등장 즉시 번역 — 각 용어를 생활어로 치환했거나 같은 자리에서 `용어(=쉬운 뜻)`로 풀었나?
[ ] 3. 정확명 보존 — 사용자가 원문을 알아야 할 때만 원어를 `원어`(=쉬운 뜻) 형태로 함께 남겼나?
[ ] 4. Tier 판단 — 단순 보고/요약이면 인라인 풀이로 끝냈나? 새 개념 소개일 때만 7요소를 썼나?
[ ] 5. 초심자 재독 — 프로젝트를 모르는 사람이 결과와 그 중요성을 한 번에 이해하나?
```

## 반례 (적용하지 않을 때)

- 식별자 / 수학 기호가 든 코드 블록
- CI 머신 파이프 JSON / JSONL 출력
- 내러티브 없는 순수 코드 출력
- 사유가 명시된 긴급 보안 경보 (심각도 정당화 시 5개 이모지 허용)
